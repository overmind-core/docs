Test datasets are the ground truth the Overmind Optimizer runs your agent against. Each dataset is a JSON array of input/expected\_output pairs: you provide the inputs your agent will receive and the outputs it should produce, and Overmind uses these pairs to score every candidate during the optimization loop. You can bring your own data, import an existing JSON file, or let Overmind generate synthetic cases from your policy — all three paths produce a versioned dataset stored in the console.

## Dataset format

Each datapoint in your dataset is a JSON object with two keys: `input` and `expected_output`. The shape of these objects matches your agent's interface — whatever your agent receives as input and returns as output.

Here is a complete example using a lead qualification agent:

```json
[
  {
    "input": {
      "company_name": "Acme Corp",
      "employee_count": 1200,
      "inquiry": "Need enterprise pricing for our sales team"
    },
    "expected_output": {
      "category": "hot",
      "lead_score": 85,
      "reasoning": "Large enterprise requesting custom pricing indicates high intent"
    }
  },
  {
    "input": {
      "company_name": "SmallBiz LLC",
      "employee_count": 15,
      "inquiry": "Just browsing your features page"
    },
    "expected_output": {
      "category": "cold",
      "lead_score": 20,
      "reasoning": "Small company with no specific intent or product interest"
    }
  }
]
```

<Tip>
  **10–50 diverse cases is typically enough.** You don't need a large dataset to get meaningful improvements. Focus on covering the main success paths, common failure modes, and your most important edge cases. If you don't have data yet, Overmind synthesizes cases from your policy and agent description — see [Synthetic data generation](#synthetic-data-generation) below.
</Tip>

## Where to put data files

Place your JSON dataset file in a `data/` directory under your agent project root:

```
your-agent-project/
├── agent.py
├── data/
│   └── test_cases.json
└── .overmind/
```

Overmind finds files in `data/` automatically. You can also provide the path explicitly when `/overmind-generate-spec-and-dataset` asks for it.

## Using existing data

When you run `/overmind-generate-spec-and-dataset`, the skill asks whether you have existing data. If you do, provide the path to your JSON file:

```
/overmind-generate-spec-and-dataset my-agent
> Do you have existing test data? yes
> Path to your data file: data/test_cases.json
```

The assistant reconciles field names in your data file with your agent's actual interface before importing, so minor naming differences are resolved before anything is pushed to the backend.

## Synthetic data generation

If you don't have data, the skill generates synthetic test cases automatically from your policy document and agent description. Generated cases cover the main scenarios from your policy's **Decision Rules** and **Edge Cases** sections, giving the optimizer a realistic and diverse test suite from the start.

You can review the generated cases before they are saved. After approval, they are pushed to the console as a versioned dataset. You can supplement or replace synthetic data with real cases at any time by running `/overmind-generate-spec-and-dataset` again.

## Dataset versioning

Each run of `/overmind-generate-spec-and-dataset` creates a new dataset version in the console. The **active version** is the one the optimizer uses during the next optimization run. You can manage versions — view, activate, or delete them — from the Datasets tab at [console.overmindlab.ai/agents](https://console.overmindlab.ai/agents).

This versioning model means you can safely iterate on your dataset without losing previous versions, and you can roll back to an earlier dataset if a new version produces unexpected results.

<Note>
  **Train/holdout split is automatic.** Overmind automatically holds out a portion of your dataset before optimization begins. The held-out cases are never seen during the optimization loop — they are used only for a final overfitting check after the best agent is selected. You never need to manage this split manually.
</Note>

## Scoring dimensions

During each optimization iteration, Overmind scores every agent output against the expected output across the following dimensions:

| Dimension | What it checks |
|---|---|
| **Structural correctness** | Are all expected fields present and non-empty in the agent's output? |
| **Value accuracy** | Do enum fields match the expected value? Are numeric fields within the configured tolerance band? |
| **Cross-field consistency** | Do related output fields agree with each other — for example, does a `"hot"` category correspond to a `lead_score` ≥ 70? |
| **Tool usage** | Were the right tools called, with correct parameters, in the correct order? |
| **LLM-as-Judge** *(optional)* | A judge model provides semantic scoring using policy-aware rubrics for dimensions that cannot be checked mechanically, such as reasoning quality or response tone. |

All scoring weights and tolerance bands are configured in the eval spec, which is derived from your policy and editable from the console.
