Tracing is how Overmind sees what your application actually does at runtime. Call `init()` once in Python or `initTracing()` once in JavaScript/TypeScript, and every supported LLM call is captured and exported to the Overmind dashboard automatically. On top of automatic capture, the Python SDK gives you decorators and span helpers to trace your own functions, tools, and workflows — turning a flat list of LLM calls into a structured tree you can read, debug, and optimize. Tracing is also the fuel for the Optimizer: the more real traces a project has, the more signal Overmind has to improve your prompts and pick better models.

## How tracing works

Overmind tracing is built on [OpenTelemetry](https://opentelemetry.io/). You don't need to know OTel to use it, but the mental model helps:

- A **trace** is one end-to-end operation — for example, handling a single request.
- A trace is a tree of **spans**. Each span represents one unit of work — an LLM call, a tool invocation, a function, a loop iteration — and records a start time, duration, status, and attributes.
- Spans **nest**. A `workflow` span can contain `tool` spans, which contain `llm_call` spans. That parent/child structure is what the dashboard renders as a timeline.

There are two layers you can use, and most applications use both:

1. **Automatic instrumentation** — after `init()`, calls to supported providers (OpenAI, Anthropic, Gemini, Agno) are traced with zero code changes.
2. **Manual instrumentation** — decorators and context managers you add to your own code to capture the surrounding business logic, not just the raw model calls.

Everything is exported over OTLP to `https://api.overmindlab.ai` and appears in your dashboard in near real time.

## Python quick start

<Steps>
  <Step title="Install the SDK">
    Install `overmind` alongside the LLM provider you use:

    ```bash
    pip install overmind openai
    ```
  </Step>

  <Step title="Initialize once at startup">
    Call `init()` a single time at process startup, **before** any LLM calls or framework setup — at the top of `main.py`, `app.py`, or your worker entry point:

    ```python
    from overmind import init

    init(
        service_name="my-service",
        environment="production",
        providers=["openai"],
    )
    ```
  </Step>

  <Step title="Call your LLM as normal — it's automatically traced">
    Keep using your LLM client exactly as before. No import swaps, no proxy routing:

    ```python
    from openai import OpenAI

    client = OpenAI()
    response = client.chat.completions.create(
        model="gpt-5-mini",
        messages=[{"role": "user", "content": "Explain quantum computing"}],
    )
    ```

    Every provider call automatically becomes an `llm_call` span capturing prompts, completions, model parameters, token usage, latency, and errors.
  </Step>
</Steps>

## JavaScript / TypeScript quick start

<Steps>
  <Step title="Install the SDK">
    Install `@overmind-lab/trace-sdk` alongside your provider package:

    ```bash
    npm install @overmind-lab/trace-sdk openai
    ```
  </Step>

  <Step title="Create an OvermindClient">
    Instantiate the client with your API key and a service name:

    ```typescript
    import { OvermindClient } from "@overmind-lab/trace-sdk";

    const overmindClient = new OvermindClient({
      apiKey: process.env.OVERMIND_API_KEY!,
      appName: "my-app",
    });
    ```
  </Step>

  <Step title="Call initTracing()">
    Pass your imported provider module to wire it up for automatic instrumentation:

    ```typescript
    import { OpenAI } from "openai";

    overmindClient.initTracing({
      enableBatching: true,
      enabledProviders: { openai: OpenAI },
    });
    ```

    All subsequent calls to your LLM clients are now traced automatically.
  </Step>

  <Step title="Shut down before exit">
    Call `shutdown()` to flush any buffered spans before your process exits — especially important for scripts and serverless handlers:

    ```typescript
    await overmindClient.shutdown();
    ```
  </Step>
</Steps>

## Tracing your own code (Python)

Automatic instrumentation only sees the LLM calls. To understand _why_ a call happened — which tool ran, which workflow it belonged to, what the inputs were — you wrap your own code in spans. There are three approaches, from highest-level to lowest:

<Tabs>
  <Tab title="Decorators">
    Decorators are the easiest way to trace a whole function. Apply one above any function definition and Overmind opens a span when the function is called, captures inputs and outputs, records duration, and closes the span when the function returns or raises.

    ```python
    from overmind import init, entry_point, workflow, tool

    init(service_name="research-agent", providers=["openai"])

    @tool("web_search")
    def web_search(query: str) -> list[str]:
        ...

    @workflow("research")
    def research(topic: str) -> str:
        results = web_search(topic)
        ...

    @entry_point("handle_request")
    def handle_request(topic: str) -> str:
        return research(topic)
    ```

    Because `handle_request` calls `research`, which calls `web_search`, the spans nest into a single tree: `handle_request` → `research` → `web_search` → the auto-traced `llm_call` spans underneath.

    Use `@observe` when you want a custom name and an explicit type in one place:

    ```python
    from overmind import observe, SpanType

    @observe("score_candidate", type=SpanType.FUNCTION)
    def score(candidate: str) -> float:
        ...
    ```

    All decorators work on both sync and async functions.
  </Tab>

  <Tab title="start_span()">
    When a decorator doesn't fit — a loop body, an `if` branch, a sub-section of a larger function — use the `start_span()` context manager. It opens a child span under the currently active span and closes it when the `with` block exits:

    ```python
    from overmind import start_span, set_tag, SpanType

    for i, doc in enumerate(documents):
        with start_span(
            "process_document",
            span_type=SpanType.FUNCTION,
            attributes={"doc.index": i},
        ) as span:
            result = process(doc)
            set_tag("doc.tokens", result.tokens)
    ```

    Unlike decorators, `start_span()` does not auto-capture inputs and outputs — you attach exactly the metadata you want via the `attributes` argument or `set_tag()` inside the block.
  </Tab>

  <Tab title="get_tracer()">
    For spans whose lifetime can't be expressed as a single block — one that starts in one callback and ends in another — drop down to the raw OpenTelemetry tracer via `get_tracer()`:

    ```python
    from overmind import get_tracer

    tracer = get_tracer()

    # Recommended: context manager form activates the span so children nest correctly
    with tracer.start_as_current_span("manual_step") as span:
        span.set_attribute("step", "load")
        ...

    # Fully manual: you are responsible for calling span.end()
    span = tracer.start_span("long_lived_op")
    try:
        span.set_attribute("phase", "begin")
        do_work()
    finally:
        span.end()
    ```

    `get_tracer()` raises `RuntimeError` if you call it before `init()`. Prefer `@observe` or `start_span()` unless you specifically need manual lifecycle control.
  </Tab>
</Tabs>

## Span types

Every Overmind span has a **type** that drives how the dashboard categorizes and renders it. Import `SpanType` to reference the enum values:

```python
from overmind import SpanType
```

| `SpanType` | Attribute value | Meaning |
|---|---|---|
| `SpanType.ENTRY_POINT` | `entry_point` | The top of a request or run — your outermost traced function |
| `SpanType.WORKFLOW` | `workflow` | A multi-step process or pipeline |
| `SpanType.TOOL` | `tool_call` | A tool or function an agent invokes |
| `SpanType.FUNCTION` | `function` | A plain traced function (the default) |
| `SpanType.LLM` | `llm_call` | A model call (set automatically by auto-instrumentation) |

## Context helpers

These helpers operate on the **current** span — call them inside a traced function, a `start_span()` block, or a request handler.

### `set_user()`

Tag the active trace with a user identity, ideally once per request in middleware:

```python
from overmind import set_user

set_user(user_id="user-123", email="alice@example.com", username="alice")
```

Only `user_id` is required. `email` and `username` are optional.

### `set_tag()`

Attach arbitrary key/value metadata you can filter on in the dashboard. Strings, numbers, booleans, and `list[str]` are stored natively; richer values are JSON-encoded automatically:

```python
from overmind import set_tag

set_tag("workflow", "order-processing")
set_tag("tenant.id", tenant_id)
set_tag("retry.count", 2)
```

### `capture_exception()`

Record an exception on the current span and mark it as an error. Use it in `except` blocks where you handle the failure but still want it visible in the trace:

```python
from overmind import capture_exception

try:
    result = client.chat.completions.create(...)
except Exception as e:
    capture_exception(e)
    raise
```

The decorators and `start_span()` already do this for exceptions that propagate out of them — `capture_exception()` is for failures you catch and swallow.

## Best practices

<Tip>
  Follow these guidelines to get the most out of Overmind tracing:

  - **Call `init()` / `initTracing()` once, first.** Place it before any LLM call or framework setup, at the top of your entry point.
  - **Name services per agent or workflow.** Give the support bot, the summarizer, and the code assistant distinct `service_name` / `appName` values so Overmind keeps their agents separate.
  - **Trace the work around the model call, not just the call.** Wrap your entry point with `@entry_point`, multi-step logic with `@workflow`, and each tool with `@tool` — that structure is what makes traces and the optimizer useful.
  - **Tag traces with the metadata you'll want to filter on.** Use `set_user()` for identity and `set_tag()` for tenant, feature flag, workflow, or version.
  - **Enable batching in production.** Use `enableBatching: true` in JS/TS and call `force_flush_traces()` or `shutdown()` before short-lived processes exit.
  - **Roll out in staging first, then production.** Continuous real-world traces give the optimizer the most signal to work with.
</Tip>
