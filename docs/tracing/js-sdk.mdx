The Overmind JavaScript and TypeScript tracing SDK instruments your LLM provider calls with a single `initTracing()` call. Unlike the Python SDK — which offers decorators, `start_span()`, and a rich set of manual instrumentation helpers — the JS/TS SDK focuses on automatic provider instrumentation: wire up your existing OpenAI, Anthropic, or Google Gemini clients once and every subsequent call is captured and exported to Overmind without any changes to your call sites. For custom spans in JavaScript or TypeScript, use the standard OpenTelemetry API directly.

## Installation

Install `@overmind-lab/trace-sdk` alongside the provider package you use:

<CodeGroup>
  ```bash OpenAI
  npm install @overmind-lab/trace-sdk openai
  ```

  ```bash Anthropic
  npm install @overmind-lab/trace-sdk @anthropic-ai/sdk
  ```

  ```bash Google Gemini
  npm install @overmind-lab/trace-sdk @google/genai
  ```
</CodeGroup>

## Quick starts

<Tabs>
  <Tab title="OpenAI">
    ```typescript
    import { OpenAI } from "openai";
    import { OvermindClient } from "@overmind-lab/trace-sdk";

    const overmindClient = new OvermindClient({
      apiKey: process.env.OVERMIND_API_KEY!,
      appName: "my app",
    });

    overmindClient.initTracing({
      enableBatching: false,
      enabledProviders: { openai: OpenAI },
    });

    const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

    const response = await openai.chat.completions.create({
      model: "gpt-5-mini",
      messages: [{ role: "user", content: "Explain quantum computing" }],
    });

    await overmindClient.shutdown();
    ```
  </Tab>

  <Tab title="Anthropic">
    ```typescript
    import * as Anthropic from "@anthropic-ai/sdk";
    import { OvermindClient } from "@overmind-lab/trace-sdk";

      appName: "my app",

    const message = await client.messages.create({

    ```

## Using multiple providers

Pass all the providers you want to instrument in a single `initTracing()` call:

```typescript
import { OpenAI } from "openai";
import * as Anthropic from "@anthropic-ai/sdk";
import * as GoogleGenAI from "@google/genai";
import { OvermindClient } from "@overmind-lab/trace-sdk";

const overmindClient = new OvermindClient({
  apiKey: process.env.OVERMIND_API_KEY!,
  appName: "my app",
});

overmindClient.initTracing({
  enableBatching: true,
  enabledProviders: {
    openai: OpenAI,
    anthropic: Anthropic,
    googleGenAI: GoogleGenAI,
  },
});
```

## `OvermindClient` configuration

Instantiate `OvermindClient` once and pass it the options below. The instance is the entry point for both `initTracing()` and `shutdown()`.

<ParamField path="apiKey" type="string" required>
  Your Overmind API key. Falls back to the `OVERMIND_API_KEY` environment variable.
</ParamField>

<ParamField path="appName" type="string" optional>
  Name of your service, shown in the dashboard. Defaults to `"overmind-js"`.
</ParamField>

<ParamField path="baseUrl" type="string" optional>
  Override the Overmind ingest endpoint. Falls back to the `OVERMIND_TRACES_URL` environment variable, then `https://api.overmindlab.ai`.
</ParamField>

---

## `initTracing()` options

<ParamField path="enabledProviders" type="object" required>
  Pass one or more imported provider modules to instrument. See the supported keys table below.
</ParamField>

<ParamField path="enableBatching" type="boolean" optional>
  Set to `true` to batch spans before export — recommended for production to reduce network overhead. Set to `false` to export immediately (useful during local development to see traces right away). Defaults to `false`.
</ParamField>

<ParamField path="instrumentations" type="Instrumentation[]" optional>
  Additional OpenTelemetry instrumentations to register alongside the provider instrumentations.
</ParamField>

<ParamField path="spanProcessors" type="SpanProcessor[]" optional>
  Additional span processors, such as custom exporters, to add to the tracing pipeline.
</ParamField>

### Supported `enabledProviders` keys

| Key | Import | Provider |
|---|---|---|
| `openai` | `import { OpenAI } from "openai"` | OpenAI |
| `anthropic` | `import * as Anthropic from "@anthropic-ai/sdk"` | Anthropic |
| `googleGenAI` | `import * as GoogleGenAI from "@google/genai"` | Google Gemini |

| `DEPLOYMENT_ENVIRONMENT` | Tag traces with an environment (e.g. `production`, `staging`). Defaults to `development`. |

| `ANTHROPIC_API_KEY` | Your Anthropic API key |

## What gets traced

For each enabled provider, the SDK automatically captures:

- Prompts and completions (messages, contents, tool calls)
- Model name, temperature, top-p, and max tokens
- Token usage (prompt tokens, completion tokens, total)
- Latency per request
- Errors and exceptions

All data is attached to OpenTelemetry spans and exported to Overmind.

---

## Resource attributes

Every trace is tagged with the following attributes automatically:

| Attribute | Value |
|---|---|
| `service.name` | Value of `appName` |
| `service.version` | SDK version |
| `deployment.environment` | `DEPLOYMENT_ENVIRONMENT` env var or `"development"` |
| `overmind.sdk.name` | `overmind-js` |
| `overmind.sdk.version` | SDK version |

## Shutdown

Call `shutdown()` to flush any buffered spans and cleanly stop the OpenTelemetry SDK before your process exits. This is especially important when `enableBatching: true` is set, because buffered spans may not have been exported yet.

```typescript
await overmindClient.shutdown();
```

For scripts or serverless handlers, use a `try/finally` block to guarantee `shutdown()` is always called:

```typescript
try {
  // ... your LLM calls
} finally {
  await overmindClient.shutdown();
}
```

For long-running servers, hook shutdown into your process exit signal:

```typescript
process.on("SIGTERM", async () => {
  await overmindClient.shutdown();
  process.exit(0);
});
```

## Production recommendations

<Tip>
  - **Enable batching in production.** Set `enableBatching: true` to reduce network overhead. Use `enableBatching: false` during local development to see traces immediately.
  - **Always call `shutdown()` before exit.** For scripts and serverless handlers, wrap your code in `try/finally`. For servers, handle `SIGTERM`. Without `shutdown()`, buffered spans from a batched pipeline may be lost.
</Tip>
