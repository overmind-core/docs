The Overmind Python tracing SDK automatically instruments your LLM provider calls and gives you decorators, context managers, and metadata helpers to trace the business logic around those calls. Install it once, call `init()` at startup, and every supported provider call is captured and exported to your Overmind dashboard — no import changes, no proxy routing required.

## Installation

Install the core SDK:

```bash
pip install overmind
```

Install alongside your LLM provider:

```bash
pip install overmind openai          # OpenAI
pip install overmind anthropic       # Anthropic
pip install overmind google-genai    # Google Gemini
pip install overmind agno            # Agno
```

## `init()`

Call `init()` once at application startup, before any LLM calls or framework setup. After calling it, your existing LLM client code is automatically instrumented with no further changes.

```python
from overmind import init

init(
    overmind_api_key="ovr_...",
    service_name="my-service",
    environment="production",
    providers=["openai"],
)
```

`init()` is idempotent and re-entrant — calling it again with additional `providers` enables those providers without tearing down the existing tracer.

### Parameters

<ParamField path="overmind_api_key" type="str" optional>
  Your Overmind API key. Falls back to the `OVERMIND_API_KEY` environment variable.
</ParamField>

<ParamField path="service_name" type="str" optional>
  Name of your service, shown in the dashboard. Falls back to the `OVERMIND_SERVICE_NAME` environment variable, then defaults to `"overmind-telemetry"`.
</ParamField>

<ParamField path="environment" type="str" optional>
  Deployment environment, such as `"production"` or `"staging"`. Falls back to the `OVERMIND_ENVIRONMENT` or `ENVIRONMENT` environment variable, then defaults to `"development"`.
</ParamField>

<ParamField path="providers" type="list[str]" optional>
  Which LLM providers to instrument. Supported values: `"openai"`, `"anthropic"`, `"google"`, `"agno"`. Pass `None` or an empty list `[]` to auto-detect and instrument all installed providers.
</ParamField>

<ParamField path="overmind_base_url" type="str" optional>
  Override the Overmind ingest endpoint. Falls back to the `OVERMIND_API_URL` environment variable, then `https://api.overmindlab.ai`.
</ParamField>

### Environment variables

| Variable | Description |
|---|---|
| `OVERMIND_API_KEY` | Your Overmind API key |
| `OVERMIND_SERVICE_NAME` | Service name (overridden by the `service_name` parameter) |
| `OVERMIND_ENVIRONMENT` | Environment name (overridden by the `environment` parameter) |
| `OVERMIND_API_URL` | Custom API endpoint URL |

## Resource attributes

Every span produced by the Python SDK is tagged with the following resource attributes automatically:

| Attribute | Value |
|---|---|
| `service.name` | Value of `service_name` (or `OVERMIND_SERVICE_NAME` env var, defaults to `"overmind-telemetry"`) |
| `service.version` | `SERVICE_VERSION` env var, otherwise `"unknown"` |
| `deployment.environment` | Value of `environment` (or `OVERMIND_ENVIRONMENT` / `ENVIRONMENT` env var, defaults to `"development"`) |
| `overmind.sdk.name` | `overmind-python` |
| `overmind.sdk.version` | SDK version |

## Decorators

Decorators are the easiest way to trace a function. The SDK ships one general decorator, `@observe`, plus four typed shortcuts:

| Decorator | Equivalent to | Span type |
|---|---|---|
| `@observe(span_name=None, type=SpanType.FUNCTION)` | — | Configurable via `type=` |
| `@entry_point(name=None)` | `@observe(type=SpanType.ENTRY_POINT)` | `entry_point` |
| `@workflow(name=None)` | `@observe(type=SpanType.WORKFLOW)` | `workflow` |
| `@tool(name=None)` | `@observe(type=SpanType.TOOL)` | `tool_call` |
| `@function(name=None)` | `@observe(type=SpanType.FUNCTION)` | `function` |

Every decorator opens a span named after the function (or the name you pass), captures positional and keyword arguments as the `inputs` attribute and the return value as `outputs` (both JSON-serialized, best-effort), records duration, sets `overmind.status` to `success`, and on exception records the exception, sets the status to `failed`, and re-raises. All decorators work on both sync and async functions.

```python
from overmind import init, entry_point, workflow, tool, function, observe, SpanType

init(service_name="research-agent", providers=["openai"])

@tool("web_search")
def web_search(query: str) -> list[str]:
    ...

@workflow("research")
def research(topic: str) -> str:
    results = web_search(topic)
    ...

@entry_point("handle_request")
def handle_request(topic: str) -> str:
    return research(topic)

# @function is equivalent to @observe(type=SpanType.FUNCTION)
@function("score_candidate")
def score(candidate: str) -> float:
    ...

# Use @observe when you want a custom name and an explicit type together
@observe("preprocess", type=SpanType.FUNCTION)
def preprocess(text: str) -> str:
    ...

# Async is supported transparently
@tool("fetch_user")
async def fetch_user(user_id: str) -> dict:
    return await db.get_user(user_id)
```

<Note>
  Call decorators with parentheses — `@workflow()` or `@workflow("name")`. If you omit the name, the function name is used as the span name.
</Note>

## `start_span()`

When a decorator doesn't fit — a loop body, an `if` branch, or a sub-section of a larger function — use the `start_span()` context manager. It opens a child span under the currently active span and closes it when the `with` block exits.

### Parameters

<ParamField path="name" type="str" required>
  Span name shown in the dashboard.
</ParamField>

<ParamField path="span_type" type="SpanType" optional>
  The span's type. Defaults to `SpanType.FUNCTION`.
</ParamField>

<ParamField path="attributes" type="dict[str, Any]" optional>
  Attributes stamped on the span at start. Rich values (dicts, Pydantic models) are JSON-encoded automatically. Defaults to `None`.
</ParamField>

```python
from overmind import start_span, set_tag, SpanType

for i, doc in enumerate(documents):
    with start_span(
        "process_document",
        span_type=SpanType.FUNCTION,
        attributes={"doc.index": i},
    ) as span:
        result = process(doc)
        set_tag("doc.tokens", result.tokens)
```

Unlike decorators, `start_span()` does not auto-capture inputs and outputs — attach exactly the metadata you want via the `attributes` argument or `set_tag()` inside the block. The context manager yields the underlying OpenTelemetry span if you need it.

## `get_tracer()`

Get the OpenTelemetry `Tracer` instance for creating custom spans around arbitrary code blocks. Use this for the rare case where a span's lifetime genuinely can't be expressed as a single `with` block.

```python
from overmind import init, get_tracer
from openai import OpenAI

init(service_name="pipeline")

client = OpenAI()
tracer = get_tracer()

def summarise_document(doc_id: str, text: str) -> str:
    with tracer.start_as_current_span("summarise") as span:
        span.set_attribute("doc.id", doc_id)
        span.set_attribute("doc.length", len(text))

        response = client.chat.completions.create(
            model="gpt-5-mini",
            messages=[
                {"role": "system", "content": "Summarise the following document concisely."},
                {"role": "user", "content": text},
            ],
        )
        return response.choices[0].message.content
```

<Note>
  `get_tracer()` raises `RuntimeError` if you call it before `init()`. Returns an `opentelemetry.trace.Tracer` instance.
</Note>

## `PromptString`

`PromptString` lets you declare a prompt's template and dynamic inputs explicitly, so Overmind can reliably match calls to the right agent across providers and versions — rather than guessing from raw strings.

```python
from opentelemetry.overmind.prompt import PromptString
from openai import OpenAI
import overmind
import uuid

overmind.init(service_name="support-bot", providers=["openai"])

system_prompt = PromptString(
    id="support_greeter_v1",
    template="You are a helpful support agent. Greet the user by name. Your name is {agent_name}.",
    kwargs={"agent_name": "Astra"},
)

client = OpenAI()
response = client.chat.completions.create(
    model="gpt-5-mini",
    messages=[
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": "How do I reset my password?"},
    ],
)
```

The `id` field (e.g. `support_greeter_v1`) gives Overmind a stable signal for agent discovery, keeping all versions of a given agent grouped and version comparisons clean. Separating `template` from `kwargs` lets Overmind identify the static template precisely and treat only the placeholders as dynamic inputs.

**Usage rules:**
- Each LLM call accepts at most one `PromptString`. The SDK raises an error if it detects more than one in a single call.
- Using the same `PromptString.id` across OpenAI, Anthropic, and Gemini lets Overmind group them as the same agent even when the raw strings differ slightly.
- Start with your most important agents and let Overmind infer the rest from plain strings.

## `set_user()`

Associate the current trace with a user identity. Call this once per request — typically in middleware — so all LLM calls made during that request are tagged with the user's identity.

```python
from overmind import set_user

set_user(user_id="user-123", email="alice@example.com", username="alice")
```

### Parameters

<ParamField path="user_id" type="str" required>
  Unique user identifier.
</ParamField>

<ParamField path="email" type="str" optional>
  User's email address.
</ParamField>

<ParamField path="username" type="str" optional>
  User's display name.
</ParamField>

### FastAPI middleware example

```python
from fastapi import FastAPI, Request
from overmind import set_user

app = FastAPI()

@app.middleware("http")
async def add_user_context(request: Request, call_next):
    user = getattr(request.state, "user", None)
    if user:
        set_user(user_id=user.id, email=user.email)
    return await call_next(request)
```

## `set_tag()`

Add a custom key-value attribute to the current span. Use this to attach any metadata you want to appear alongside traces in the dashboard. Strings, numbers, booleans, and `list[str]` are stored natively; richer values (dicts, dataclasses, Pydantic models) are JSON-encoded automatically.

```python
from overmind import set_tag

set_tag("feature.flag", "new-checkout-flow")
set_tag("tenant.id", tenant_id)
set_tag("workflow", "order-processing")
```

### Parameters

<ParamField path="key" type="str" required>
  Attribute name.
</ParamField>

<ParamField path="value" type="str" required>
  Attribute value.
</ParamField>

## `capture_exception()`

Record an exception on the current span and mark its status as `ERROR`. Use this in `except` blocks where you want the trace to reflect the failure even when you handle or swallow the exception.

```python
from overmind import capture_exception

try:
    result = client.chat.completions.create(...)
except Exception as e:
    capture_exception(e)
    raise
```

### Parameters

<ParamField path="exception" type="Exception" required>
  The exception to record on the current span.
</ParamField>

<Note>
  The decorators and `start_span()` already call `capture_exception()` automatically for exceptions that propagate out of them. Use this helper only for failures you catch and handle yourself.
</Note>

## Span attributes reference

Every span produced by SDK helpers carries a consistent set of attributes:

| Attribute | Set by | Description |
|---|---|---|
| `overmind.span.type` | All span helpers | One of the `SpanType` values |
| `overmind.status` | All span helpers | `success`, `failed`, or `cancelled` (on `KeyboardInterrupt`) |
| `overmind.duration.seconds` | All span helpers | Wall-clock duration of the span |
| `overmind.error.type` / `overmind.error.message` | On failure | Exception class name and a truncated, scrubbed message |
| `inputs` / `outputs` | Decorators only | JSON-serialized arguments and return value |
| `user.id` / `user.email` / `user.username` | `set_user()` | Current user identity |
| _(your keys)_ | `set_tag()` / `attributes=` | Any custom metadata you attach |

## Full example

The following example wires together `@entry_point`, `@tool`, `start_span`, `set_user`, `set_tag`, and `capture_exception` into a single customer-support trace:

```python
import os
from overmind import (
    init, entry_point, tool, start_span,
    set_user, set_tag, capture_exception, SpanType,
)
from openai import OpenAI

os.environ.setdefault("OVERMIND_API_KEY", "ovr_your_key_here")

init(service_name="customer-support", environment="production", providers=["openai"])

client = OpenAI()

@tool("lookup_order")
def lookup_order(order_id: str) -> dict:
    return {"id": order_id, "status": "shipped"}

@entry_point("handle_support_query")
def handle_support_query(user_id: str, question: str) -> str:
    set_user(user_id=user_id)
    set_tag("workflow", "support")

    with start_span("gather_context", span_type=SpanType.FUNCTION):
        order = lookup_order("A-1001")

    try:
        response = client.chat.completions.create(
            model="gpt-5-mini",
            messages=[
                {"role": "system", "content": "You are a helpful customer support agent."},
                {"role": "user", "content": f"{question}\n\nOrder: {order}"},
            ],
        )
        return response.choices[0].message.content
    except Exception as e:
        capture_exception(e)
        raise

print(handle_support_query("user-123", "Where is my order?"))
```

This produces a single trace: `handle_support_query` (entry point) → `gather_context` → `lookup_order` (tool) → the OpenAI `llm_call`, all tagged with the user and workflow.

## Advanced helpers

The following helpers live in `overmind.tracing` and cover less common cases — import them explicitly:

```python
from overmind.tracing import (
    start_child_span,    # like start_span(), but force-nests under the current span
    set_workflow_name,   # label every downstream span with a workflow name
    set_agent_name,      # bind downstream spans to a named agent
    set_conversation_id, # group multiple traces into one chat session
    conversation,        # decorator form of set_conversation_id
    observe_safe,        # like @observe, but skips input/output capture (for sensitive data)
    force_flush_traces,  # best-effort flush before a short-lived process exits
)
```

`observe_safe` is useful for functions that handle secrets or large payloads you don't want serialized into span attributes. Call `force_flush_traces()` at the end of scripts and serverless handlers so buffered spans are exported before the process dies.
